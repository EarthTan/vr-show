## 3. **Critical environment note** (read this before doing anything)

The plan was authored with `wgpu = "26"` and `egui = "0.32"`. **Those versions do not build on this machine**. Task 1 already corrected this; the workspace currently uses:

```toml
# Cargo.toml [workspace.dependencies]
wgpu = "27"      # was 26
egui = "0.33"    # was 0.32
egui-wgpu = "0.33"
egui-winit = "0.33"
# MSRV bumped from 1.85 to 1.88
```

**Why**: `naga 26.0.0` has a `codespan-reporting` trait-bound regression that breaks compilation on rustc ≥ 1.86. Reference: [`gfx-rs/wgpu#7968`](https://github.com/gfx-rs/wgpu/issues/7968) — closed as duplicate, fix is in `naga 27+` / `wgpu 27+`. The fix `#9243` was merged 2026-03-16.

**The host system**: rustc 1.95.0 / cargo 1.95.0. Compatible with everything chosen.

**If you see a `naga 26.0.0 ... WriteColor` error during any subsequent task**, it means `Cargo.lock` got partially regenerated and pinned to naga 26. Fix with:

```bash
rm Cargo.lock && cargo build
```

This was the only blocker encountered — once you have a working `Cargo.lock`, no further issues are expected for the remaining tasks.

## 5. Workflow to resume

**Use the `subagent-driven-development` skill** (or `executing-plans` if you want a parallel session). Read the plan, dispatch implementer subagent + spec-reviewer + code-reviewer per task.

**Three hints for whoever picks this up:**

1. **Do Task 4 (camera tests + camera state) first**, not Task 3. Camera state is referenced by App state (Task 13) so defining it early is cleaner. You can re-order 3 ↔ 4 if desired; the plan says 3 first but says "scene::sphere and scene::texture don't exist yet, so add empty placeholders next" in Task 4 — meaning Task 4 already imports Task 6 files (texture.rs). Look at Task 4's Step 3 — it explicitly creates empty `sphere.rs` and `texture.rs` files. So you can't do Task 4 *truly* first without Task 6 — just do Task 3 then 4 in order, with the placeholders as 4's Step 3 says.

2. **For Task 9 (renderer), the plan's snippet for `egui-wgpu` API may need adjustment**. wgpu 27's `DeviceExt::create_buffer_init` is at `wgpu::util::DeviceExt`. The plan already imports this via `use wgpu::util::DeviceExt as _` — keep that pattern; it's fine. If you hit API mismatches on `egui-wgpu::Renderer::new`, the signature in 0.33 is:
    ```rust
    egui_wgpu::Renderer::new(
        device: &wgpu::Device,
        output_format: wgpu::TextureFormat,
        depth_format: Option<wgpu::TextureFormat>,
        msaa_samples: u32,
    ) -> Self
    ```
    The plan's Task 16 calls a longer form that may not exist in 0.33 — adjust based on `cargo doc -p egui-wgpu --open` if needed.

3. **For Task 16 (egui integration), do not try to set up the whole integration in one shot**. Split it: first do "create egui context + render empty frame with `lib.rs.run(...)` clearing the egui pass" and verify visually. Then layer in the empty state, HUD, error banner separately. This task has the highest uncertainty about exact API surface.

## 7. Known risks remaining

1. **Linux build dependencies**: The workspace uses `winit` with `wayland`, `wayland-dlopen`, `wayland-csd-adwaita` features. On Linux this requires system libraries at build time (not at runtime). The user already has Tauri 2 working on Linux, so the system likely has them, but if `cargo build` fails on Linux with link errors about `wayland-client-0`, those packages need installing. The existing `TESTING.md` already documents the apt-get command.

2. **macOS Gatekeeper** for unsigned builds — already covered in the existing Tauri section of `TESTING.md`, applies equally to Rust build.

3. **Test image dimensions**: `Qwen-Image-2512_00001_.png` is 2048×1024 (verified in Task 7's test). If a future manual smoke test exposes different dimensions, the aspect-ratio warning is logged and rendering still works — but if the file is removed, regenerate or substitute any 2:1 PNG.
